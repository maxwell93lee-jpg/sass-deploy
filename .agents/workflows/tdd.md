# TDD (Test-Driven Development) 가이드 및 프롬프트

AI 어시스턴트(또는 개발자)가 본 프로젝트에서 새로운 기능을 구현하거나 코드를 수정할 때 반드시 준수해야 하는 TDD 원칙과 프롬프트 지침입니다. 코드를 작성하기 전에 항상 이 문서를 기준으로 삼아주세요.

## 🎯 핵심 TDD 사이클

모든 개발은 다음 세 가지 단계를 엄격하게 순서대로 진행해야 합니다:

1. **🔴 Red Phase (실패하는 테스트 작성)**
   - 실제 비즈니스 로직이나 UI 코드를 작성하기 **전에**, 요구사항을 검증하는 테스트 코드를 먼저 작성합니다.
   - 이 시점에서는 구현체가 없거나 불완전하므로 테스트가 실패해야 정상입니다.

2. **🟢 Green Phase (테스트 통과를 위한 최소한의 구현)**
   - 실패한 테스트를 통과시키기 위한 **가장 단순하고 최소한의 코드**만 작성합니다.
   - 복잡한 로직이나 완벽한 설계보다는 오직 "테스트 통과"에 집중합니다.

3. **🔵 Refactor Phase (코드 개선 및 리팩토링)**
   - 테스트가 통과하는 안전한 상태(Green)에서 코드의 구조를 개선합니다.
   - 중복을 제거하고, 가독성을 높이며, `클린 아키텍처` 및 프로젝트 설계 규칙(`Next.js-framework.md`)에 맞게 로직을 다듬습니다. 리팩토링 후에도 모든 테스트는 여전히 통과해야 합니다.

## 📋 테스트 종류 및 작성 기준

단일 기능에 대해 다음 세 가지 계층의 테스트를 모두 고려하여 작성해야 합니다:

- **Unit Test (단위 테스트)**: 독립적인 함수, 도메인 엔티티(Entities), 개별 UI 컴포넌트, 분리된 유스케이스(Use Cases)의 동작을 검증합니다.
- **Integration Test (통합 테스트)**: 여러 모듈 간의 상호작용(예: Repository 구현체가 DB와 통신하는 과정, 상태 관리와 UI의 결합)을 검증합니다.
- **E2E Test (End-to-End 테스트)**: 실제 사용자의 관점에서 로그인부터 결제까지 핵심 비즈니스 흐름 전체가 올바르게 작동하는지 검증합니다.

## 🚀 필수 요구사항 및 강제 규칙 (Prompts)

AI가 코드를 제안하거나 수정할 때는 다음 프롬프트 명령을 내재화하여 스스로 수행해야 합니다:

1. **커버리지 방어선**: "모든 코드 변경 후 프로젝트의 전체 **테스트 커버리지(Test Coverage)를 80% 이상**으로 유지해줘."
2. **최종 검증 로직**: "Red-Green-Refactor를 거쳐 모든 구현이 끝나면, **반드시 전체 테스트 스위트를 다시 실행(Run All Tests)**해서 기존 기능을 포함한 모든 테스트가 통과하는지 확인해줘."
3. **자체 치유(Self-Correction) 프로세스**: "전체 테스트 실행 중 통과하지 못한(Failed) 테스트가 단 하나라도 존재한다면, **왜 통과하지 못했는지 근본 원인을 자세히 분석해서 설명하고, 해당 원인을 해결하도록 코드를 수정하여 다시 테스트를 통과시켜 줘.**"

---
**적용 요약**: 이 룰을 기반으로 작업할 때는 답변에 항상 `[Red]`, `[Green]`, `[Refactor]` 단계를 명시하여 어떤 흐름으로 코드를 도출했는지 설명하고, 마지막에 전체 테스트 통과 여부 및 커버리지에 대한 코멘트를 포함해 주세요.
